# 30 邮箱按时间同步迭代记录

日期：2026-06-10

## 需求来源

用户反馈“同步邮件数量感觉不是很具体”，希望改成按时间同步。该反馈符合真实邮箱管理场景：用户通常更关心“最近几天/某个时间段内有哪些邮件需要处理”，而不是抽象地同步最近 N 封。

## 本轮目标

1. 邮箱同步从“数量优先”改为“时间范围优先”。
2. 保留最多封数作为安全上限，避免真实邮箱一次读取过多邮件。
3. 支持常用快捷范围和自定义日期范围。
4. 后端 IMAP 查询真正使用时间条件，而不是前端伪筛选。
5. 每次同步刷新本地邮件快照，避免新旧邮件混在同一个列表中。

## 已完成

- 前端同步区新增“同步时间”下拉：
  - 今天。
  - 最近 3 天。
  - 最近 7 天。
  - 最近 30 天。
  - 自定义范围。
- 自定义范围支持选择开始日期和结束日期。
- “最多封数”保留为安全上限，默认 50，最大 200。
- 后端 `ImapSyncRequest` 新增：
  - `since_date`
  - `until_date`
- 后端 `ImapSyncRequest` 使用 `replace_snapshot=true` 作为默认行为：IMAP 拉取成功后清空本地旧邮件、Todo、草稿、附件记录和 Agent 日志，再写入本次同步结果。
- IMAP 服务将日期转换为 `SINCE` / `BEFORE` 查询条件。
- 结束日期按用户直觉做包含当天处理，内部通过 `BEFORE end_date + 1 day` 实现。
- 邮件 `Date` 头会被归一化为 `YYYY-MM-DD HH:MM:SS`，前端列表按该时间倒序排列。
- 后端会在 IMAP 初筛后根据解析出的本地日期再次过滤，减少“选择今天却看到其他日期邮件”的情况。
- 账号记忆会同步记住时间范围选择。

## 验收点

1. 默认打开页面时，同步范围为“最近 7 天”。
2. 选择“自定义范围”后，显示开始日期和结束日期。
3. 点击“按时间同步”时，请求体包含 `since_date` 和 `until_date`。
4. 后端 OpenAPI 中 `/api/emails/sync` 包含时间字段。
5. 同步成功后，左侧邮件列表只显示本次同步结果，不混入上一次同步的旧邮件。
6. 原有 IMAP 同步、邮件列表刷新、后续 Agent 处理流程不受影响。
